# Docs-sync quality and scope — design

**Status:** design only. No implementation in this doc.
**Predecessors:** [`2026-07-16-docs-sync-design.md`](2026-07-16-docs-sync-design.md), [`2026-07-16-docs-sync-implementation.md`](2026-07-16-docs-sync-implementation.md).
**Not published:** this file lives under `docs/plans/`, which is outside the docs-site loader allowlist (`docs-site/src/loaders/repo-docs.ts`), so it never renders to customers.

## Goal

Make the prose that docs-sync writes good enough to ship to **customers** without a human rewriting it, make the set of docs it touches match the set the docs site actually publishes, and fix the quality of what is already live.

The docs site (`https://docs.opslane.com`) is a customer surface. It is read by three people, in order of how much a bad doc costs us:

1. **A prospect evaluating Opslane.** Sloppy or wrong docs read as a sloppy product. This is the cheapest place to lose a deal, and we never see it happen.
2. **A new customer integrating the SDK.** If the React snippet doesn't run verbatim, their smoke test fails, they file a ticket or churn. Docs are onboarding.
3. **An AI assistant** answering "how do I use Opslane?" from `llms.txt` (generated by `starlight-llms-txt` over every published page, not just the navigable ones). Wrong prose here misinforms prospects we never talk to.

The bar is not "technically accurate." It is: **a prospect who follows this page verbatim succeeds, and comes away trusting us.** Everything below is measured against that bar.

## Why now

The v1 pipeline works end-to-end (proved on a live PR). Three things are unbuilt, all customer-visible:

- **The prompt is thin and unmeasured.** `promptForDocument` (`scripts/docs-sync/plan.mjs`) is five lines. On the dogfood PR it produced a clean edit to `guides/react.md` and, on the same run, echoed schema-wrapper junk into two architecture docs. We hardened the *pipeline* to reject malformed output (#85). We did not make the *edits themselves* reliably good, and there are zero evals.
- **The scope model is wrong on paper.** The v1 design and the first draft of this doc equated "public" with the site sidebar. That is incorrect (see the model below), and it hides real gaps.
- **The baseline is unaudited.** The incremental pipeline only maintains quality *from here*. Nobody has run the published snippets end-to-end or asked whether an internals page needs a diagram. So this doc also plans a one-time baseline pass (C6).

## Non-goals

- **Not** changing the delivery model. docs-sync keeps committing to the source PR branch; edits stay human-reviewed before merge, before the site.
- **Not** touching the deterministic `reference/` gate. `check-docs-drift.mjs` keeps routes, env vars, SDK options, and reason codes correct. LLM prose editing stays away from it.
- **Not** a docs rewrite as an ongoing behavior. C6 is a one-time baseline; steady state is minimal incremental edits.

## The publication model (three distinct sets)

"Public" is defined by the **loader allowlist**, not the sidebar. `docs-site/src/loaders/repo-docs.ts` admits `install.md` plus everything under `quickstart/`, `guides/`, `reference/`, `architecture/`, and `contracts/`. The sidebar (`docs-site/astro.config.mjs`) is only navigation and lists a hand-picked subset. Concretely: the loader publishes all three of `contracts/reliability.md`, `contracts/C4-amendments.md`, and `contracts/events.md`, but the sidebar names only the first two — so `events.md`, a normative wire contract, is live and in `llms.txt` yet unreachable by clicking.

Model three sets explicitly and stop conflating them:

- **P — published pages.** The loader allowlist. What a customer or `llms.txt` can actually reach.
- **N — navigable pages.** The sidebar slugs. A subset of P chosen for the nav.
- **Policy(page).** For every page in P, exactly one docs-sync policy: **prose** (LLM-synced, guides/architecture/quickstart/install), **deterministic** (drift-gated, reference), **manual** (hand-maintained, e.g. contracts unless C5 moves them), or **excluded**.

`docs/plans/**`, this doc included, is intentionally **not** in P — the loader even resolves symlinks before its allowlist check (`repo-docs.ts` `createCanonicalEnricher`) so a committed symlink can't leak a plan into the site.

## Two kinds of doc, plus contracts

The prose tier is not one thing. It splits by reader intent, and the split lines up with how the site is organized. A classifier — `docTypeOf(path)` next to `isProseTierDoc` in `scripts/docs-map.mjs` — returns `setup | internals | contract`. C1 injects type-specific guidance; C2 scores each type against its own bar. One canonical definition, so the prompt, the evals, and a reviewer don't each guess.

**Setup — "get me running."** `guides/**`, `quickstart/**`, `install.md`. Read by a customer mid-integration.
- **Success:** they copy, paste, and it works.
- **Medium:** runnable snippets and exact commands. A renamed import here is the highest-cost bug in the system — an onboarding failure we never see.
- **Voice:** imperative, step-by-step. Edit shape is usually a snippet or command, not a paragraph.

**Internals — "how does it work, can I trust it."** `architecture/**`. Read by a prospect or engineer deciding whether to trust the system with production errors.
- **Success:** an accurate mental model they believe.
- **Medium:** diagrams and data flow over code. The site renders Mermaid (`astro-mermaid`); when a diff changes a pipeline stage, permission, or data path, update the **diagram and its caption**, not one sentence.
- **Voice:** explanatory but plain — no jargon. Scrutinize `trust.md` and `precision.md` hardest.

**Contract — "the exact, normative promise."** `contracts/**` (only if C5 pulls them into the prose tier). A reliability or wire contract (`events.md`) is normative: append-only, precise, "must/never" language, and a source of truth other systems depend on.
- **Bar is different:** no "update the diagram and caption" guidance; no loosening or paraphrasing normative wording; every change requires stronger human review than an ordinary prose edit. If we cannot meet that bar with an LLM edit, contracts stay **manual** (C5).

## The customer bar, made concrete

An edit is acceptable to publish only if a reader could not tell a machine wrote it:

- **Runnable snippets run.** Every fence classified `runnable` (see C3's snippet contract) executes verbatim against the current SDK in a fixture app. A wrong import or renamed export in a runnable snippet is a P0. Non-runnable fences (templates, fragments, output) are held to their own class, not to "runs verbatim."
- **Says only what the diff proves.** No invented options, defaults, or behaviors. Over-claiming in customer docs is worse than silence.
- **Readable, no jargon.** Plain language a smart non-expert follows on the first read. Technical terms that name a real thing are fine — SDK, error boundary, source map, webhook — but define an unavoidable one in a short clause and never lean on buzzwords or insider shorthand. Prefer the plainer word ("safe to run twice," not "idempotent").
- **Right medium for the type.** Setup wants a snippet or command; internals wants a diagram and a data path; a contract wants exact normative text.
- **Keeps the page's voice.** Terse, second-person, confident, no hedging. No "it is important to note that…" scaffolding.
- **Minimal.** Touch the stale span, not the paragraph around it. Small diffs are reviewable; large ones get rubber-stamped, and that is how bad prose reaches prospects.

## Design

### C1 — Prompt hardening

Rewrite `promptForDocument` around the customer bar, branching on `docTypeOf(docPath)`:

- **Customer framing.** State that the output ships to a public docs site read by prospects and used to onboard paying customers, and that a runnable snippet which does not run is the worst outcome.
- **Type-specific guidance.** Setup → optimize for a snippet or command that runs verbatim. Internals → an accurate Mermaid diagram and data path plus the sentence around it. Contract → preserve normative wording exactly; never paraphrase a "must"/"never"; smallest possible change.
- **Readable, no jargon.** Write for a smart non-expert; define any unavoidable term in a short clause; no buzzwords.
- **Minimal-edit discipline.** Change the smallest span that makes the page true. Do not restructure, re-order, retitle, or reflow untouched prose. Return the document byte-for-byte identical if nothing is stale.
- **Voice preservation.** Match surrounding tense, person, and terseness.
- **Medium fidelity.** Setup: fences stay valid and consistent with the diff's real symbol names. Internals: Mermaid stays parseable and consistent with the described flow.
- **Structured self-report.** Keep `{content, changed}`; reinforce that wrapper tags never appear in `content` (the #85 guard is the backstop, not the first line of defense).

The prompt stays a pure function of `(docPath, original, diff)` — `docType` is derived from `docPath` — so it is fully reproducible in evals (C2). **C1 does not land until C2's baseline exists** (see Rollout): no prompt change ships on "looks better."

### C2 — Two separate test tiers

The current mocked tests and a future prompt eval are different things and must not be conflated.

**C2a — Deterministic pipeline and validator tests (CI-required).** A mocked `runClaude` verifies staging, rejection, skip-on-`changed:false`, schema-tag rejection, and every C3 gate. These are cheap, hermetic, and already the pattern in `scripts/__tests__`. They prove the *plumbing*, never the *prose*.

**C2b — Live prompt-quality evaluation (offline, not per-PR).** Calls the real pinned CLI against a fixture corpus and scores the result. To be meaningful it needs, all specified up front:
- **Pinned model, not just the package.** `plan.mjs` pins `@anthropic-ai/claude-code@2.1.211` but passes no `--model`; the resolved model can drift and masquerade as a prompt change. Pin an explicit model (and, if a second LLM judges output, pin the judge model too) as a prerequisite to any eval claim. This pinning should also land in production `claudeArgs()`.
- **A corpus of `(original doc, code diff) → expectations`.** Seed from the real #83 cases: the `react.tsx → react.md` edit that worked, and the two architecture docs that should have returned `changed:false`. Only **code-change-induced** cases belong here (see C6/point below).
- **Exact metrics and thresholds, per type.** Shared: stale-fix rate, false-edit rate (the #83 failure), diff size, readability/jargon. Setup adds runnable-snippet-passes and command validity. Internals adds Mermaid-parses and diagram-matches-prose. Contract adds normative-wording-preserved. Each metric has a numeric promotion threshold defined *before* the eval is used as a gate.
- **A repeated-run policy.** LLM output is non-deterministic; specify N runs per case and how variance is scored (e.g. worst-of-N for safety metrics, mean for quality metrics).
- **Stored baseline reports.** Each run writes a dated report; a prompt change is judged against the stored baseline, not a vibe. Golden/expected results are exact strings or explicit required/forbidden-edit assertions, not "reads better."

C2b is advisory until its thresholds are trusted, then it gates prompt-file changes only.

### C3 — Edit quality gates, in two stages

Validation splits by what context it needs. Do not pretend a link check is content-local.

**Stage 1 — content-local gates (per edited file, in `plan.mjs` before staging).** Deterministic, single-file: frontmatter (esp. `covers:`) preserved; schema-wrapper rejection (#85); code fences balanced and language-tagged; Mermaid blocks parse; snippet-contract conformance (below); diff-size policy (below).

**Stage 2 — repo-context / site gates (in trusted publish, over the whole checkout).** Internal-link integrity and a site-realistic Mermaid render need the full repo with **all** proposed edits overlaid — a link can only be validated against the other pages that exist. The link validator is an Astro **build plugin** (`starlight-links-validator`), not a standalone single-file API, so "run it against the edited doc" is not viable; it must run as a build over the overlaid checkout.

The current publish path copies files then immediately commits and pushes (`scripts/docs-sync/publish.mjs`). Respecify the trusted publish sequence:
1. Re-run Stage-1 content-local gates on each staged file.
2. Apply all edits into the PR-head checkout.
3. Run the Stage-2 repo-context/site validation (a docs-site build, or an equivalent link+diagram check over the overlaid tree).
4. Commit and guarded-push **only if Stage 2 passes**; otherwise fail the job, push nothing.

**Snippet contract.** "Runs verbatim" is only meaningful for snippets that are supposed to run. Setup docs contain runnable commands, compilable fragments, config templates with placeholders (`install.md`: `VITE_OPSLANE_API_KEY=<your Opslane ingest key>`), expected-output blocks, and illustrative pseudo-code. Classify every fence as `runnable | fragment | config-template | expected-output | illustrative`, via a fence annotation or a small per-doc manifest that maps `runnable` fences to fixture files and commands (`test-fixtures/react-app`, `vue-app`). Enforceable criterion: **every `runnable` snippet passes; every non-runnable snippet is explicitly classified.** An unclassified fence in a setup doc is itself a failure.

**Diff-size — one policy.** Flag for reviewer attention first (advisory), and promote to a hard block once the threshold is validated against real edits. Stated once here; acceptance and open-questions defer to this.

### C4 — Scope integrity lint

Encode the three-set model so it can't drift, using the loader as the source of truth for P:

- Every page in **P** (loader allowlist) has an explicit `Policy(page)` — prose, deterministic, manual, or excluded. A published page with no declared policy fails the build (that is how `events.md` silently sits unmanaged today).
- Every sidebar slug in **N** resolves to a page in **P**. A nav entry pointing at a non-published or missing page fails.
- **Do not** require P == N. Publishing a page without navigating to it (or vice versa) is allowed, but must be a deliberate, recorded choice, not an accident.
- The prose-policy set equals `isProseTierDoc`; deterministic equals the `reference/` set `check-docs-drift.mjs` owns; the rest are manual/excluded and named explicitly.

### C5 — Resolve scope and the contract question (do this first)

C1 and C2 both depend on `docTypeOf` and the policy map, so **C5 is resolved before C1/C2 are implemented.** Decisions:

- **Do contracts join the prose tier?** They are published (loader admits `contracts/**`) but currently synced by neither docs-sync nor drift, so `reliability.md`, `C4-amendments.md`, and `events.md` can silently rot on a customer-facing, security-relevant surface. Two acceptable resolutions: (a) add a **contract** doc type with normative-language rules and stronger review (C1/C2 support it); or (b) keep contracts **manual** and add a deterministic staleness reminder/mapping (a lighter sibling of the drift gate) so drift is at least detected. Pick per contract if needed.
- **What does `contracts/**` mean?** All of it — including `events.md` — not just the two sidebar entries. Whatever policy C5 picks applies to every contract file the loader publishes.

### C6 — Baseline audit of what's already public (one-time)

C1–C5 keep docs from drifting forward; none of them improve a page that shipped thin, wrong, or jargon-heavy. Before we lean on the site as a sales and onboarding surface, do one deliberate human-reviewed pass over everything in **P**. AI does the first-pass audit and drafts.

**Step 1 — Inventory and reconcile.** Enumerate P (loader allowlist), N (sidebar), and Policy(page). Three buckets: published-and-tiered (audit them), published-but-unmanaged (`events.md`, and any page missing a policy — resolve via C4/C5), and unpublished-but-present (`plans/`, internal — confirm intentional). This is C4 run once by hand to seed the automated version.

**Step 2 — Audit each published page against its type's bar.**
- *Setup:* run every `runnable` snippet and command against the current SDK in `test-fixtures/react-app`/`vue-app`; classify every other fence. Check for missing steps a real integrator hits and a real "verify it works."
- *Internals:* check the mental model is accurate and current; flag where a Mermaid diagram would land the concept faster, or where an existing diagram is now wrong; scrutinize `trust.md`/`precision.md` hardest.
- *Contract:* verify normative claims still match the code and the wire fixtures; flag any drift for manual correction.
- *All:* jargon/readability sweep and voice-consistency check.

**Step 3 — Prioritized gap report.** One artifact, ranked by customer cost: P0 onboarding-blocker (a runnable snippet that fails), P1 trust/accuracy error in an internals or contract doc, P2 missing diagram or step, P3 readability/voice. Each item names the doc, the reader it hurts, and the fix.

**Step 4 — Remediate; seed evals selectively.** Fix P0/P1 by hand as per-doc PRs (not one mega-commit). **Only gaps caused by a specific code change become C2b fixtures** — those have a real `(original, code diff)` and test exactly what docs-sync is for. Baseline defects with no causal diff (a jargon-heavy sentence nobody's code changed) do **not** go into C2b — feeding them in would train docs-sync to make unrelated rewrites, breaking the minimal-edit rule. Those live in a separate static-audit / page-level regression suite.

## Human review is the customer gate

The auto-committed edit is a **proposal**. The PR reviewer is the last line before a customer sees it. C1–C3 exist to make that proposal small, correct, and in-voice so the reviewer can judge it in seconds — not to remove the human. A proposal too big or too weird to review in seconds should have been caught by C3's size/link/diagram gates.

## Acceptance criteria

- `docTypeOf` returns `setup | internals | contract`, and C5's contract policy is encoded in the policy map (C4/C5).
- C4 lint fails when a published page (P) has no declared policy, or a sidebar slug (N) does not resolve to a published page; it does **not** require P == N.
- C2a (deterministic) is CI-required and covers the #83 skip/reject cases. C2b (live) runs offline with a pinned model + judge, stored baseline reports, defined per-type thresholds, and a repeated-run policy; no prompt lands without moving C2b metrics against baseline.
- A setup edit's `runnable` snippets pass in a fixture app and every non-runnable fence is classified; an internals edit that changed a shown flow updates the Mermaid diagram and keeps it parseable; a contract edit preserves normative wording.
- Trusted publish runs Stage-1 content-local gates, overlays edits, runs the Stage-2 build/link/diagram validation, and pushes only on pass.
- Diff-size uses the single policy in C3 (flag first, block after threshold validated) — consistently in acceptance and open questions.
- C6 has run: every published `runnable` snippet executes, a ranked gap report exists, P0/P1 are fixed, and only causal-diff gaps became C2b fixtures.

## Verification

- C2a green in CI; C2b produces a dated, stored report and compares to baseline with exact golden results or required/forbidden-edit assertions (not "byte-identical-quality to a hand edit").
- C4 lint fails on a seeded policy gap (drop `events.md`'s policy) and on a seeded dangling sidebar slug; passes on the real tree.
- Stage-1 gates reject seeded bad edits (dropped frontmatter, broken fence, unclassified runnable snippet, oversized diff); Stage-2 build rejects a seeded broken internal link and a broken Mermaid diagram.
- Dogfood: re-run the `react.tsx` change and assert the edit matches the golden result; run a contract-touching change once C5's policy is set.

## Rollout

1. **P0 — scope truth.** C5 decision (contracts) + C4 model and lint. Cheap; unblocks and encodes what everything else keys off. Includes C6 Step 1 inventory by hand.
2. **P1 — measurement before change.** Pin the model in `claudeArgs()`; build C2a and a minimal C2b from the #83 cases; capture a baseline report. No prompt edits yet.
3. **P2 — trust the output.** C1 prompt hardening + C3 two-stage gates, each change measured against P1's baseline.
4. **P3 — fix the baseline.** C6 Steps 2–4; expand C2b only with causal-diff gaps; route readability defects to the static-audit suite.

## Open questions

- `contracts/**`: prose tier with a normative contract type, or manual with a deterministic staleness reminder? (C5 — needs a call, blocks P0.)
- Live evals (C2b): run nightly, or only on prompt-file changes? Cost vs. drift-catch.
- Readability/jargon check: heuristic (banned-phrase list, sentence-length delta) or a pinned second-LLM judge? Start heuristic; escalate only if it misses.
- Stage-2 validation cost: a full docs-site build per publish may be slow — is a scoped link+Mermaid check over the overlaid tree enough, or do we accept the build?
